# 1. 개발 도구 역할 재정의

## 1) Obsidian: 프로젝트 컨트롤타워

Obsidian은 코드 실행 도구가 아니라, **프로젝트의 사고·결정·실험·리서치 결과를 쌓는 지식베이스**로 쓰는 게 좋습니다.

추천 역할은 다음과 같습니다.

|영역|Obsidian 역할|
|---|---|
|프로젝트 개요|목표, 범위, 원칙, 금지사항 정리|
|개발 로드맵|4단계와 하위 Phase 관리|
|투자 철학|가치투자 기준, 매매 원칙, 리스크 원칙 정리|
|데이터 명세|사용할 데이터 소스, 컬럼, 검증 기준 정리|
|실험 기록|스크리너 기준 변경, 백테스트 결과, 실패 기록|
|리서치 결과|AI Berkshire 분석 요약, 최종 투자 점수표|
|의사결정 로그|왜 이 기준을 채택/폐기했는지 기록|

즉, Obsidian은 **개발 노트 + 투자 운영 매뉴얼 + 실험 로그**입니다.

## 3) Codex CLI: 메인 개발자

Codex CLI는 다음을 담당시키는 게 좋습니다.

|담당|예시|
|---|---|
|프로젝트 구조 생성|폴더, 설정 파일, 패키지 구조|
|데이터 수집 코드|KRX, DART, 가격 데이터 수집 모듈|
|정량 스크리너 구현|ROE, FCF, 부채비율, 거래대금 필터|
|점수화 로직 구현|정량 점수 + AI 리서치 점수 통합|
|백테스트 구현|룩어헤드 방지, 수수료, 슬리피지 반영|
|매매 규칙 구현|ATR 손절, 트레일링 스탑, 시장 국면 필터|
|테스트 작성|단위 테스트, 데이터 검증 테스트|
|리팩터링|모듈 분리, 타입 정리, 문서화|

Codex는 Skills를 통해 반복 작업을 패키징할 수 있고, 공식 문서 기준으로 Skill은 `SKILL.md`를 포함한 디렉터리이며 `name`과 `description` 메타데이터가 필요합니다. 또한 CLI/IDE/Codex app에서 사용할 수 있고, `$`로 명시 호출하거나 설명 기반으로 자동 선택될 수 있습니다.

다만 이 프로젝트에서는 처음부터 Codex Skill을 너무 많이 만들기보다, **AGENTS.md + 단계별 작업 프롬프트 + 필요한 시점의 Skill** 정도로 시작하는 게 낫습니다.

## 4) Claude Code: 리서치 전담

Claude Code는 실제 코드 개발보다는 **AI Berkshire를 활용한 투자 리서치 전담 도구**로 쓰는 편이 좋습니다.

AI Berkshire는 18개 Skill 개요를 제공하고, `/investment-team`은 4개 Agent가 병렬로 기업을 분석한 뒤 Team Lead가 종합하는 구조라고 설명합니다. Claude Code 공식 문서도 slash command가 `/`로 시작하는 특수 명령이고, custom command를 만들 수 있다고 설명합니다.

따라서 Claude Code 담당은 다음처럼 제한하는 게 좋습니다.

|담당|사용 예|
|---|---|
|AI Berkshire 리서치|`/investment-team 삼성전자`|
|개별 기업 정성 분석|사업모델, 해자, 경영진, 리스크|
|투자 논리 검증|Bull/Bear, 반대 논리, 실패 시나리오|
|보유 논리 추적|thesis tracker|
|리서치 리포트 산출|Markdown 보고서 생성|

Claude Code는 리서치 결과를 `reports/research/`에 저장하고, Codex가 그 결과를 읽어 점수화/정리/통합하는 구조가 좋습니다.


# 2. 최종 개발 구조 재정의

이제 전체 구조는 아래 4단계로 고정하면 됩니다.

```
Stage 1. 정량적 스크리너국내 전체 종목 → 후보 30개Stage 2. AI 리서치후보 30개 → AI Berkshire식 분석 → 투자 가치 점수화Stage 3. 최종 선별상위 5~10개 → 관심 universe 등록Stage 4. 매매 시스템기술적 진입 조건 + ATR 손절 + 트레일링 스탑 + 시장 국면별 비중 조절
```

중요한 점은 **AI 리서치가 매매 신호를 직접 만들지 않는 것**입니다. AI 리서치는 “투자 가능한 종목군을 고급화하는 필터”이고, 실제 매매는 별도 규칙으로 처리해야 합니다.
