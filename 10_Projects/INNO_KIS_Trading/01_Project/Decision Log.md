## 2026-06-27 - 프로젝트를 Stage 0부터 시작하기로 결정

### 배경

국내 주식 대상 투자 리서치 및 자동매매 시스템을 개발하기 전에, 코드부터 작성하면 데이터 원칙, AI 사용 범위, 매매 규칙, 리서치 기준이 섞일 위험이 있다.

특히 이 프로젝트는 아래 요소가 함께 포함된다.

- 정량 스크리너
- AI 리서치
- 관심 universe 관리
- 기술적 매매 시스템
- ATR 손절
- 트레일링 스탑
- 시장 국면별 비중 조절

따라서 본격 개발 전에 프로젝트 원칙과 구조를 먼저 정의할 필요가 있다.

### 선택지

1. 바로 Python 코드 개발을 시작한다.
2. GitHub repo만 먼저 만들고 개발하면서 문서를 작성한다.
3. Stage 0에서 프로젝트 원칙, 문서, repo 구조, AI 사용 규칙, 데이터 정책을 먼저 정의한다.

### 결정

선택지 3을 따른다.

Stage 0에서 프로젝트 기반을 먼저 구축한 뒤 Stage 1 정량 스크리너 개발로 넘어간다.

### 이유

- AI 리서치와 매매 판단의 경계를 명확히 할 수 있다.
- 금융 데이터를 임의 생성하지 않는 원칙을 초기에 고정할 수 있다.
- raw data 수정 금지 원칙을 초기에 고정할 수 있다.
- Codex CLI와 Claude Code의 역할을 분리할 수 있다.
- config 기반 설계를 초기에 반영할 수 있다.
- 추후 백테스트에서 룩어헤드 바이어스와 과최적화를 줄일 수 있다.

### 리스크

- 초기 문서 작성에 시간이 걸린다.
- 실제 코드 개발 시작이 늦어진다.
- 문서가 과하게 복잡해질 수 있다.

### 후속 작업

- Project Charter.md 작성
- Development Roadmap.md 작성
- System Architecture.md 작성
- Decision Log.md 작성
- GitHub repo 구조 생성
- AGENTS.md 작성
- CLAUDE.md 작성
- config/*.yaml 설계
- 데이터 저장 정책 정의

## 2026-06-27 - AI Berkshire를 매수/매도 판단기가 아닌 리서치 전용 프레임워크로 사용하기로 결정

### 배경

AI Berkshire는 기업의 질적 분석에는 유용할 수 있지만, AI가 직접 매수/매도 판단을 내리게 하면 책임 소재와 검증 가능성 문제가 발생한다.

이 프로젝트는 자동매매 시스템을 포함하지만, AI 리서치 결과가 직접 매매 신호가 되어서는 안 된다.

### 선택지

1. AI Berkshire가 매수/매도 의견까지 작성한다.
2. AI Berkshire가 투자 점수와 매수/매도 신호를 함께 생성한다.
3. AI Berkshire는 정성 리서치와 투자 가치 점수화까지만 수행하고, 매매 판단은 규칙 기반 시스템으로 분리한다.

### 결정

선택지 3을 따른다.

AI Berkshire는 리서치 전용 프레임워크로만 사용한다.

### 이유

- AI의 추정과 실제 매매 규칙을 분리할 수 있다.
- 매수/매도 판단의 자동화를 방지할 수 있다.
- 리서치 결과를 universe 고급화 필터로만 사용할 수 있다.
- 실제 진입과 청산은 검증 가능한 규칙 기반 시스템으로 관리할 수 있다.

### 리스크

- 리서치 점수가 높아도 실제 매매 조건이 발생하지 않을 수 있다.
- 리서치 결과와 기술적 진입 조건이 충돌할 수 있다.
- 정성 점수화 기준의 일관성 관리가 필요하다.

### 후속 작업

- CLAUDE.md에 AI Berkshire 사용 규칙 작성
- 리서치 리포트 템플릿 작성
- research_score.yaml 설계
- 매수/매도 지시 금지 문구 명시

## 2026-06-27 - 모든 스크리너 및 매매 기준을 config에서 관리하기로 결정

### 배경

스크리너 기준과 매매 기준을 코드 안에 직접 작성하면, 실험 재현성과 기준 변경 이력이 약해진다.

따라서 기준값은 config 파일에서 관리하고, 코드는 config를 읽어 실행하는 구조로 만든다.

### 선택지

1. 기준값을 Python 코드 안에 직접 작성한다.
2. 주요 기준값만 config로 분리한다.
3. 스크리너, 리서치 점수, 매매 규칙, 리스크 기준, 데이터 출처를 모두 config로 분리한다.

### 결정

선택지 3을 따른다.

아래 config 파일을 사용한다.

- config/universe.yaml
- config/screener.yaml
- config/research_score.yaml
- config/trading_rules.yaml
- config/risk.yaml
- config/data_sources.yaml

### 이유

- 실험 재현성이 높아진다.
- 기준 변경 이력을 관리하기 쉽다.
- 코드와 전략 기준을 분리할 수 있다.
- 과최적화 여부를 추적하기 쉽다.
- Stage별 책임이 명확해진다.

### 리스크

- config 파일이 복잡해질 수 있다.
- config 값 검증 로직이 필요하다.
- 잘못된 config 입력 시 잘못된 결과가 나올 수 있다.

### 후속 작업

- config 파일 초안 작성
- config schema 검증 방식 검토
- tests에서 config 로딩 테스트 작성
